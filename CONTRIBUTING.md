# Contributing to Survival Shooter Game

Thank you for your interest in contributing! This document provides guidelines for contributing to this project.

## ü§ù How to Contribute

### Reporting Bugs
- Use GitHub Issues for bug reports
- Include browser version, OS, and steps to reproduce
- Provide screenshots if applicable
- Check existing issues before creating new ones

### Suggesting Features
- Open an issue with "Feature Request" label
- Describe the feature and its benefits
- Consider how it fits the game design
- Provide implementation ideas if you have them

### Code Contributions
1. Fork the repository
2. Create a feature branch: `git checkout -b feature-name`
3. Make your changes
4. Test thoroughly
5. Commit with descriptive messages
6. Push to your fork
7. Submit a pull request

## üõ†Ô∏è Development Guidelines

### Code Style
- Use consistent indentation (4 spaces)
- Follow existing naming conventions
- Add comments for complex logic
- Keep functions focused and small

### File Organization
- `game.js` - Core game logic only
- `sounds.js` - Audio system only
- `style.css` - All styling
- `index.html` - HTML structure

### Testing
- Test in multiple browsers (Chrome, Firefox, Safari)
- Check for performance impact
- Verify no breaking changes
- Test edge cases

## üéÆ Game Design Principles

### Balance Considerations
- Maintain fair difficulty progression
- Avoid power creep with new content
- Keep risk-reward mechanics intact
- Preserve skill expression

### Content Guidelines
- New enemies should fit existing themes
- Upgrades should be balanced across rarity tiers
- Sound effects should match game tone
- Visual style should remain consistent

## üìù Pull Request Process

### Before Submitting
- [ ] Code follows project style guidelines
- [ ] Self-review completed
- [ ] Testing performed
- [ ] Documentation updated if needed
- [ ] No console errors
- [ ] Performance impact considered

### PR Description
Include in your pull request:
- Clear description of changes
- Reasoning for the changes
- Testing performed
- Any breaking changes
- Screenshots if UI changes

### Review Process
- Maintainer will review within 48 hours
- Feedback will be provided for any issues
- Changes may be requested before merge
- Approved PRs will be merged to main
